## Successes

1. **Retrospective-to-Epic closed loop** — Epic 3 retro discovered env var deprecation → `/bmad-correct-course` → redefined Epic 4 → delivered. Proves retrospectives can directly change the roadmap, not just surface insights.

2. **Backward compatibility handled excellently** — `small_fast_model` config field preserved, `--small-fast-model` CLI flag became alias for `--haiku-model` via `sfmAlias` variable. Existing user configs and workflows untouched. New users get canonical `--haiku-model`/`--sonnet-model`/`--opus-model` flags.

3. **sfmAlias design avoided subtle semantic change** — Plan Review Round 1 for 4.2 caught that a naive `if haikuModel == ""` guard would silently change `--small-fast-model` duplicate behavior from last-wins to first-wins. The `sfmAlias` approach simultaneously satisfies AC4 (alias), AC5 (haiku-model priority), and AC11 (duplicate last-wins). Plan Review earned its value here.

4. **Epic-internal debt digestion is healthy** — 4.1 deferred integration test gap → filled in 4.2. 4.2 deferred test duplication + error messages → cleaned in 4.3. Story 4.3 itself produced zero new patches (all findings were defer). The "quality cleanup story" pattern works.

5. **Plan Review safety net consistently effective** — Every story had substantive findings caught at planning stage. TDD ordering violations caught in both 4.2 (HIGH) and 4.3 (HIGH). sfmAlias semantic issue caught in 4.2 Round 1. `--haiku-model ""` edge case identified in 4.3 Round 1.

## Challenges

1. **TDD ordering violated in Stories 4.2 and 4.3** — Both stories had implementation tasks before test tasks in initial plans. Same issue flagged in Epic 3 retro. Plan Review caught it both times, but this is the third consecutive retrospective mentioning TDD ordering. **Root cause identified this time: the `bmad-create-story` process doesn't bake in TDD ordering, so initial story structure naturally follows file-based grouping rather than RED→GREEN pairing.**

2. **Deferred items net growth continues** — Epic 3 retro noted digesting 5, adding 9, net +4. Epic 4 adds 11 more deferred items (though most were digested within the epic). Epic 3's 7 cross-epic deferred items (LookPath, SHELL validation, signal exit codes, etc.) still have no home.

3. **Plan Review rounds increasing** — Epic 1 stories typically passed in 1 round. Epic 4 stories needed 1-2 rounds with more findings. Reflects growing story complexity (signature changes, alias logic, field migrations) rather than declining plan quality. Higher complexity → more review value.

## Key Insights

1. **Fix the tool, not the person** — Three retrospectives of "remember TDD ordering" proved that exhortation doesn't work. The root cause is in `bmad-create-story` producing file-grouped task lists. The fix is to change the story creation process to default to RED→GREEN task pairs. This replaces the recurring "plan with TDD ordering" agreement with a structural solution.

2. **Epic-internal debt digestion is the right strategy** — Story 4.3 proved that a dedicated cleanup story within an epic can achieve zero new deferred items. This pattern should continue: defer within an epic, digest before epic closes.

3. **Retrospectives can directly change roadmaps** — Epic 4 wouldn't exist without Epic 3's Significant Discovery. The correct-course mechanism worked perfectly: discover → assess → replan → execute. This is a model for future retrospectives.

4. **Cross-epic deferred items need explicit digestion paths** — Epic 3's 7 items have been sitting in deferred-work.md without a trigger story or epic to absorb them. Explicit assignment (e.g., "all runner/cmd debt goes to Epic 5") prevents items from falling through cracks.
