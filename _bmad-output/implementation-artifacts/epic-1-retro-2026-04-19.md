## Successes

1. **Architecture-driven development was highly effective** — Precise templates with line-number references made implementation nearly zero-ambiguity. All 4 stories shipped smoothly.
2. **Runner struct extraction was clean** — run.go went from ~200 lines to ~30 lines. Zero logic duplication between run and exec.
3. **Code review process caught real issues** — 6 patches across 4 stories, including a panic from empty Target slice in Story 1.1.
4. **Brownfield first story carried dual role successfully** — Story 1.1 bore the natural weight of "paying debt + building foundation," clearing the path for Stories 1.2-1.4.
5. **Breaking changes applied consistently** — All 3 breaking changes (unknown arg hard error, TUI cancel exit 1, no contexts stderr+exit 1) implemented per architecture spec.
6. **Test coverage solid** — runner_test.go, args_test.go, config permission test all added proactively.

## Challenges

1. **Deferred items accumulated without systematic resolution** — 15+ items in deferred-work.md, most labeled "pre-existing." No mechanism to systematically address them across epics.
2. **TUI-related issues surfaced repeatedly but were deferred** — Fragile string comparison, race condition on cancelled flag, no panic recovery, misleading bounds check error — appeared in reviews for Stories 1.1, 1.2, and 1.3, deferred each time.
3. **cmd/*.go error handling code is duplicated** — Identical `fmt.Fprintf(os.Stderr, "Error: %v\n", err); os.Exit(1)` pattern repeated 6 times across run.go and exec.go.
4. **Story complexity distribution uneven** — Story 1.1 was heaviest (extraction + refactor + tests + breaking changes), Story 1.4 was trivial (one-line fix + test). Accepted as normal for brownfield projects.

## Key Insights

1. **Brownfield project's first story naturally carries the most weight** — It performs architecture restructuring on existing code, exposing hidden edge cases. This is expected, not a planning failure.
2. **Deferred items need a systematic digestion mechanism** — Without explicit allocation in sprint planning, deferred-work.md becomes a "black hole" (items go in, never come out).
3. **Architecture documentation quality directly determines delivery quality** — Precise templates, code examples, line-number references, and anti-patterns lists made implementation nearly mechanical.
